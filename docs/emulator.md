# Emulator

The emulator loads a `.bin` file and executes it instruction by instruction. It models the full CPU state — registers, data memory, stack pointer, instruction pointer, and all three flags — exactly as specified in the ISA.

## Usage

```sh
build/bin/emulator <program.bin> [--debug]
```

### Normal mode

```sh
build/bin/emulator build/ex/vcl/factorial.bin
```

```
Loaded 70 instruction(s) from build/ex/vcl/factorial.bin

======================================
  Finished in 430 step(s)
======================================

+--------------------------------------+
|  IP =   2   SP = 255   Z=0 S=0 C=0       |
+--------------------------------------+
|  R0 = 120 (0x78)   R1 =   5 (0x05)  |
|  R2 =   0 (0x00)   R3 =   0 (0x00)  |
|  R4 =   0 (0x00)   R5 =   0 (0x00)  |
|  R6 =   0 (0x00)   R7 =   0 (0x00)  |
+--------------------------------------+

-- Data memory (non-zero, below stack) --
  mem[  0] =  24 (0x18)
  mem[  1] =   5 (0x05)
  mem[  2] = 120 (0x78)
  ...
```

After `END` or `RET` at the top level, the emulator prints the final register state and a dump of all non-zero data-memory bytes below SP. The convention is to read the program's return value from R0.

### Debug mode

```sh
build/bin/emulator program.bin --debug
```

Every instruction prints its slot address, hex encoding, mnemonic, and the complete register state after execution:

```
[IP=  0] 0xB80C  CALL
+--------------------------------------+
|  IP =  12   SP = 254   Z=0 S=0 C=0       |
+--------------------------------------+
|  R0 =   0 (0x00)   R1 =   0 (0x00)  |
...
```

This is useful for tracing the execution of hand-written assembly or inspecting the code generated by the compiler.

## Step limit

The emulator runs for at most 100 000 steps. If that limit is reached it prints:

```
Warning: step limit reached - possible infinite loop
```

and reports the final state. Programs that deliberately loop forever will always hit this limit.

## Runtime errors

| Condition | Message |
|---|---|
| Stack overflow (PUSH when SP = 0x00) | `Stack overflow! SP would go below 0x00` |
| Stack underflow (POP when SP = 0xFF) | `Stack underflow! Pop from empty stack` |
| IP out of bounds | `IP out of bounds: N` |
| Unknown opcode | `Unknown opcode: N at IP=M` |

All of these halt execution immediately and report the final state.

## CPU state at startup

| Field | Initial value |
|---|---|
| R0–R7 | 0 |
| mem[0..255] | 0 |
| SP | 0xFF |
| IP | 0 |
| Z, S, C flags | 0 |

## API

Declared in `src/emulator.h`:

```c
#include "emulator.h"

/* Execute one instruction. Returns 1 to continue, 0 to halt. */
int execute_one(CPU *cpu, const Program *prog, int debug);

/* Run at most max_steps instructions. Returns the step count. */
int run_program(CPU *cpu, const Program *prog, int max_steps, int debug);
```

Typical usage in tests and the compiler's end-to-end test suite:

```c
CPU cpu;
cpu_reset(&cpu);
run_program(&cpu, &prog, 100000, 0);
int result = (int)cpu.reg[0];   /* return value is in R0 */
```

See `tests/test_emulator.c` for 30 unit tests and `tests/test_integration.c` for 10 end-to-end tests that assemble and run real programs.
